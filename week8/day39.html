ConceptsConceptos

1. What a "rehearsal" run is1. Qué es una "ejecución de ensayo"

A rehearsal means running the full pipeline end-to-end on fresh data, exactly as you would in production, to catch any remaining integration issues before the final run.

The difference between a test and a rehearsal: a test validates a single component in isolation. A rehearsal validates the entire sequence — does every output format feed correctly into the next input? Are column names consistent across tools? Do all tools run without crashing on real data?

Today is your dress rehearsal. Tomorrow is opening night.

Un ensayo significa ejecutar el pipeline completo de extremo a extremo sobre datos nuevos, exactamente como lo harías en producción, para encontrar cualquier problema de integración restante antes de la ejecución final.

La diferencia entre un test y un ensayo: un test valida un solo componente en aislamiento. Un ensayo valida la secuencia completa — ¿cada formato de salida alimenta correctamente la siguiente entrada? ¿Los nombres de columnas son consistentes entre las herramientas? ¿Todas las herramientas corren sin caerse con datos reales?

Hoy es tu ensayo general. Mañana es el estreno.

SQL Connection

This is like running a full database regression test suite before a major release. In Oracle, you'd run every stored procedure in sequence with representative data and verify the final result tables match expectations. Same principle here — you're running every tool in sequence and checking that the final output (Apollo contacts) matches what you'd expect from a clean run.

Conexión SQL

Esto es como ejecutar un conjunto completo de pruebas de regresión de base de datos antes de una versión importante. En Oracle, ejecutarías cada procedimiento almacenado en secuencia con datos representativos y verificarías que las tablas de resultados finales coincidan con las expectativas. Mismo principio aquí — estás ejecutando cada herramienta en secuencia y verificando que el resultado final (contactos en Apollo) coincida con lo esperado de una ejecución limpia.

2. Writing workflows/gtm_pipeline.md2. Escribiendo workflows/gtm_pipeline.md

This is the SOP — Standard Operating Procedure — for running the pipeline. It describes what the pipeline does, what inputs it needs, what each tool does and in what order, what to check at each step, and how to handle common errors.

You write this yourself. Not because Claude can't draft it — it can — but because the act of writing it proves you understand the system. If you can explain it clearly enough for a new person to follow, you understand it. If you can't, the gaps in your explanation reveal gaps in your understanding.

Claude will help you draft it. Then you edit it into your own words.

Este es el SOP — Procedimiento Operativo Estándar — para ejecutar el pipeline. Describe qué hace el pipeline, qué entradas necesita, qué hace cada herramienta y en qué orden, qué verificar en cada paso y cómo manejar errores comunes.

Lo escribís vos. No porque Claude no pueda hacerlo — puede — sino porque el acto de escribirlo demuestra que entendés el sistema. Si podés explicarlo con suficiente claridad para que una persona nueva lo siga, lo entendés. Si no podés, los vacíos en tu explicación revelan vacíos en tu comprensión.

Claude te va a ayudar a hacer un borrador. Después lo editás con tus propias palabras.

Key Insight

The WAT framework again: the workflow is the instruction set that any agent — or any human — can follow to reproduce the same outcome. A pipeline without a workflow is fragile. With one, it's transferable. You've just made your work trainable.

Idea Clave

El framework WAT otra vez: el workflow es el conjunto de instrucciones que cualquier agente — o cualquier persona — puede seguir para reproducir el mismo resultado. Un pipeline sin un workflow es frágil. Con uno, es transferible. Acabás de hacer que tu trabajo sea enseñable.
