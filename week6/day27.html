ConceptsConceptos

1. The Apify API — calling Actors programmatically1. La API de Apify — llamar Actors desde código

So far you have run Apify Actors manually through the web console. Today you go one level deeper: calling them from Python code using the Apify API. Instead of clicking "Run" in the browser, your script sends an HTTP request that starts the Actor, waits for it to finish, and retrieves the results — all automatically.

The flow is: your Python script → HTTP POST to Apify → Actor runs in the cloud → results come back as JSON → your script parses and saves them.

Hasta ahora ejecutaste Apify Actors manualmente desde la consola web. Hoy vas un nivel más profundo: llamarlos desde código Python usando la API de Apify. En vez de hacer clic en "Run" en el navegador, tu script envía una petición HTTP que inicia el Actor, espera a que termine y recupera los resultados — todo automáticamente.

El flujo es: tu script Python → HTTP POST a Apify → el Actor corre en la nube → los resultados vuelven como JSON → tu script los parsea y guarda.

SQL Connection

Calling Apify from Python is like DBLINK to a vendor-hosted stored procedure. You pass parameters in the POST body (like IN parameters), Apify executes the logic on their servers, and you get results back — just like a stored procedure returning a result set. You don't see the internals; you only control the inputs and consume the output.

Conexión SQL

Llamar a Apify desde Python es como un DBLINK a un stored procedure alojado por un proveedor. Pasás los parámetros en el cuerpo del POST (como parámetros IN), Apify ejecuta la lógica en sus servidores y te devuelve los resultados — igual que un stored procedure que retorna un result set. No ves los internos; solo controlás los inputs y consumís el output.

Key Insight

The apify-client Python package wraps all the HTTP calls for you: client.actor("actor-id").call(run_input={...}) starts the Actor, waits for it to complete, and returns the run object. Then you fetch the dataset to get results.

Idea Clave

El paquete Python apify-client envuelve todas las llamadas HTTP por vos: client.actor("actor-id").call(run_input={...}) inicia el Actor, espera a que termine y retorna el objeto run. Después fetcheás el dataset para obtener los resultados.

2. Building tools/fetch_tech_stack.py2. Construir tools/fetch_tech_stack.py

This script is a clean, reusable function: give it a domain, get back a dict of tech categories. Internally it calls the BuiltWith Actor on Apify, which tells you what technologies a website runs — their CMS, analytics, ad platforms, email tools, and more.

The output is a flat dict — something like {"analytics": "Google Analytics", "cms": "WordPress", "email": "Mailchimp"} — that can be stored as a single column in your enrichment CSV.

Three things to understand in the code when you read it:

  • The Actor call — where it starts the BuiltWith Actor with your domain as input
  • Polling / waiting — .call() blocks until the Actor finishes (the client handles this for you)
  • JSON parsing — how it extracts the top tech categories from Apify's response structure

Este script es una función limpia y reutilizable: le das un dominio y te devuelve un dict de categorías de tecnologías. Internamente llama al Actor BuiltWith en Apify, que te dice qué tecnologías usa un sitio web — su CMS, analytics, plataformas de publicidad, herramientas de email y más.

El output es un dict plano — algo como {"analytics": "Google Analytics", "cms": "WordPress", "email": "Mailchimp"} — que puede guardarse como una sola columna en tu CSV de enriquecimiento.

Tres cosas para entender en el código cuando lo leás:

  • La llamada al Actor — dónde inicia el Actor BuiltWith con tu dominio como input
  • Polling / espera — .call() bloquea hasta que el Actor termina (el cliente lo maneja por vos)
  • Parseo de JSON — cómo extrae las principales categorías de tech de la estructura de respuesta de Apify
SQL Connection

Integrating the output into your enrichment CSV as a tech_stack column is exactly like adding a computed column to a SELECT. Each row gets a value derived from a secondary lookup — the same pattern as a scalar subquery or a joined lookup table.

Conexión SQL

Integrar el output en tu CSV de enriquecimiento como columna tech_stack es exactamente como agregar una columna calculada a un SELECT. Cada fila obtiene un valor derivado de una búsqueda secundaria — el mismo patrón que una subquery escalar o una tabla de lookup unida.
