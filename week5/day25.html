1. Why scripts fail on real data — and how to handle it1. Por qué los scripts fallan con datos reales — y cómo manejarlo

The 20-row sample file you built this week was clean. Real lead lists are not. You'll encounter:

  • Missing domain values — the cell is blank
  • Invalid domains — "gmail.com", "n/a", a phone number in the domain column
  • API timeouts — the request never returns
  • API rate limits — 429 errors mid-loop
  • Unexpected JSON structure — the field you're looking for isn't in the response

Without error handling, any of these stops your entire script. One bad row out of 500 kills the whole run.

The pattern is simple: wrap the risky call in a try/except block, log what failed, and continue to the next row:

for index, row in df.iterrows():
    try:
        response = requests.get(url + row["domain"], headers=headers)
        company_size = response.json()["metrics"]["employees"]
        df.at[index, "company_size"] = company_size
    except Exception as e:
        print(f"Skipped row {index}: {e}")
        failed_rows.append(row)
        continue

After the loop, write the failures to a file: pd.DataFrame(failed_rows).to_csv("tools/failed_rows.csv", index=False). Now you can review what was skipped and decide whether to retry or remove those rows.

El archivo de muestra de 20 filas que construiste esta semana estaba limpio. Las listas de leads reales no lo están. Te vas a encontrar con:

  • Valores de dominio faltantes — la celda está en blanco
  • Dominios inválidos — "gmail.com", "n/a", un número de teléfono en la columna de dominio
  • Timeouts de API — la petición nunca devuelve respuesta
  • Límites de tasa de API — errores 429 a mitad del bucle
  • Estructura JSON inesperada — el campo que buscás no está en la respuesta

Sin manejo de errores, cualquiera de estos detiene todo el script. Una fila mala de 500 mata toda la ejecución.

El patrón es simple: envolvé la llamada riesgosa en un bloque try/except, registrá qué falló, y usá continue para pasar a la siguiente fila:

for index, row in df.iterrows():
    try:
        response = requests.get(url + row["domain"], headers=headers)
        company_size = response.json()["metrics"]["employees"]
        df.at[index, "company_size"] = company_size
    except Exception as e:
        print(f"Fila {index} omitida: {e}")
        failed_rows.append(row)
        continue

Después del bucle, escribí las fallas en un archivo: pd.DataFrame(failed_rows).to_csv("tools/failed_rows.csv", index=False). Ahora podés revisar qué se omitió y decidir si reintentar o eliminar esas filas.

SQL Connection

Oracle's EXCEPTION WHEN NO_DATA_FOUND THEN NULL is the same idea as Python's try/except — don't crash the whole procedure when one row fails. In PL/SQL you'd use EXCEPTION blocks inside a cursor loop. In Python you use try/except inside a for loop. Same intent, same structure, different syntax.

Conexión SQL

El EXCEPTION WHEN NO_DATA_FOUND THEN NULL de Oracle es la misma idea que el try/except de Python — no rompas todo el procedimiento cuando falla una fila. En PL/SQL usarías bloques EXCEPTION dentro de un bucle de cursor. En Python usás try/except dentro de un bucle for. Misma intención, misma estructura, sintaxis diferente.

2. Storing credentials safely with .env files2. Guardando credenciales de forma segura con archivos .env

Never hardcode API keys directly in your scripts. If you share the file with anyone, push it to GitHub, or email it — your key is exposed. Even a "private" repo has been accidentally made public before.

The safe pattern uses a .env file:

  1. Create a file named .env at your project root
  2. Add your key: CLEARBIT_API_KEY=your_actual_key_here
  3. Add .env to your .gitignore file so it never gets committed
  4. In your script, load it: from dotenv import load_dotenv; import os; load_dotenv(); key = os.getenv("CLEARBIT_API_KEY")

Now the key lives only in the .env file on your machine. The script just reads it at runtime. You can share the script freely — it won't work without the .env file, which stays local.

Nunca pongas API keys directamente en tus scripts. Si compartís el archivo con alguien, lo subís a GitHub o lo enviás por email — tu clave queda expuesta. Incluso un repositorio "privado" se ha hecho público accidentalmente antes.

El patrón seguro usa un archivo .env:

  1. Creá un archivo llamado .env en la raíz de tu proyecto
  2. Agregá tu clave: CLEARBIT_API_KEY=tu_clave_real_aqui
  3. Agregá .env a tu archivo .gitignore para que nunca se commitee
  4. En tu script, cargala: from dotenv import load_dotenv; import os; load_dotenv(); key = os.getenv("CLEARBIT_API_KEY")

Ahora la clave solo vive en el archivo .env en tu máquina. El script simplemente la lee en tiempo de ejecución. Podés compartir el script libremente — no va a funcionar sin el archivo .env, que se queda local.

Key Insight

A .env file is like Oracle's wallet or tnsnames.ora — it stores connection credentials outside the code, so the code itself is safe to share. This is not optional. Every professional Python project that touches an API uses this pattern.

Idea Clave

Un archivo .env es como el wallet de Oracle o el tnsnames.ora — guarda las credenciales de conexión fuera del código, para que el código en sí sea seguro de compartir. Esto no es opcional. Todo proyecto Python profesional que toca una API usa este patrón.

1 Why should you never hardcode API keys directly in a Python script? ¿Por qué nunca deberías poner API keys directamente en un script Python?
If you share the file, push it to a git repo (even a private one), or send it by email — the key is exposed. A leaked API key can be used by anyone who finds it, running up charges on your account or accessing data you don't want shared. Use .env files and keep them off version control.
Si compartís el archivo, lo subís a un repositorio git (incluso uno privado), o lo enviás por email — la clave queda expuesta. Una API key filtrada puede ser usada por cualquiera que la encuentre, generando cargos en tu cuenta o accediendo a datos que no querés compartir. 2 What Python construct prevents a script from crashing when one row fails? ¿Qué construcción de Python evita que un script crashee cuando falla una fila?
try/except — wrap the risky operation in a try block, handle the error in except, and use continue to skip to the next row. Always log what was skipped so you can investigate later. Never silently swallow errors.
try/except — envolvé la operación riesgosa en un bloque try, manejá el error en except, y usá continue para saltar a la siguiente fila. Siempre registrá qué se omitió para poder investigar después. Nunca silencies los errores sin registrarlos.
3 What is python-dotenv used for? ¿Para qué se usa python-dotenv?
Loading environment variables from a .env file into your script at runtime. You call load_dotenv() at the top of the script, then use os.getenv("KEY_NAME") to read each value. This lets you keep credentials out of your code entirely.
