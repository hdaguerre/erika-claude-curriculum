The WAT Framework — Workflows, Agents, Tools

Estimated time: ~3 hours
ConceptsConceptos

1. The full WAT picture1. El panorama completo de WAT

WAT stands for Workflows, Agents, Tools. It's an architecture for building reliable AI-powered systems by separating what AI does well (reasoning, decision-making) from what code does well (deterministic execution).

  • Workflow — a plain-language SOP (Standard Operating Procedure) stored as a Markdown file in workflows/. It describes what to accomplish, in what order, using which tools, and how to handle edge cases. Written like a briefing document.
  • Agent — that's Claude. It reads the workflow and makes intelligent decisions about how to execute it: which tool to call, what to do when something fails, when to ask a clarifying question.
  • Tools — Python scripts in tools/ that do exactly one thing reliably. API calls, data transformations, file operations. No reasoning — just execution.

WAT = Workflows, Agents, Tools (Flujos de trabajo, Agentes, Herramientas). Workflow = un SOP (procedimiento operativo estándar) en lenguaje simple almacenado como archivo markdown en workflows/. Describe qué hacer, en qué orden y con qué herramientas. Agent = Claude mismo — lee el workflow y toma decisiones inteligentes sobre la ejecución. Tools = scripts de Python en tools/ que hacen una cosa específica de manera confiable.

SQL Connection

WAT is a three-tier architecture — the same pattern you know from enterprise Oracle applications: the presentation layer (workflow = what the user wants), the application layer (agent = Claude making decisions), and the database layer (tools = deterministic execution). You've built systems like this. The vocabulary is different; the structure is the same.

Conexión SQL

WAT es una arquitectura de tres capas — exactamente como las aplicaciones empresariales de bases de datos: la capa de presentación (workflow = lo que el usuario quiere), la capa de aplicación (agente = Claude tomando decisiones), y la capa de base de datos (herramientas = ejecución determinística). Ella ya construyó sistemas así — solo con vocabulario diferente.

2. Why the separation matters2. Por qué importa la separación

When AI tries to do everything — reason AND execute AND handle errors AND manage state — accuracy drops fast. The math is unforgiving:

  • 1 step at 90% accuracy = 90% success
  • 2 steps at 90% each = 81% success
  • 5 steps at 90% each = 59% success
  • 10 steps at 90% each = 35% success

By offloading execution to deterministic Python scripts (always 100% predictable for a given input), you keep Claude focused on what it does well: reasoning, interpreting ambiguous situations, making judgment calls. The tools handle the parts that shouldn't involve probability.

Cuando la IA intenta hacer TODO (razonar + ejecutar + manejar errores + gestionar estado), la precisión cae rápido. Cinco pasos al 90% de precisión = 59% de éxito en total. Al separar el razonamiento (Claude) de la ejecución (scripts de Python), ambas capas se mantienen confiables.

Key Insight

A Python script that calls an API either works or it doesn't — there's no 90% about it. By using tools for execution and Claude for reasoning, you get the best of both: intelligent decision-making and reliable execution. That's what makes WAT different from just prompting Claude to do everything.

Idea Clave

La separación es lo que hace que el sistema sea confiable. Claude razona bien. Los scripts de Python ejecutan de manera confiable. Juntos son mejores que cualquiera por separado.
